# 🚀 开源社区贡献者从0到1实操指南

> 🎯 **面向谁**：第一次参与蓝区开源社区贡献的工程师。
>
> 📦 **示例项目**：以 [triton-ascend](https://github.com/triton-lang/triton-ascend) 为例，其它蓝区项目（[openEuler](https://gitcode.com/openeuler/)、[MindSpore](https://gitcode.com/mindspore) 等）流程一致。

**操作步骤**：

按指引提交 PC 申请：🔗 <https://wiki.huawei.com/domains/5248/wiki/20780/WIKI2026032610554934>

## 0.2 打通蓝区网络

**操作内容**：完成蓝区 PC 的网络配置，确保可以访问 GitHub、GitCode、PyPI 等公网服务。

**操作步骤**：

1. 按网络打通指引完成配置：🔗 <https://wiki.huawei.com/domains/5248/wiki/20780/WIKI2026050610983937>
2. 配置完成后，逐条执行以下命令做"网络体检"：

```bash
curl -I https://github.com      # 期望返回 HTTP/2 200 或 301
curl -I https://gitcode.com     # 期望返回 200
pip config list                  # 检查 pip 源是否已配置为蓝区可用源
```

3. 如需在蓝区和黄区之间互传文件，使用以下方式：
   - **蓝 → 黄**：push 到 GitHub 仓，再在黄区下载；小文件可邮件
   - **黄 → 蓝**：🔗 黄蓝文件同步助手 <https://wiki.huawei.com/domains/4971/wiki/82756/WIKI2026022710230136>

**易发问题**：

- **`curl https://github.com` 超时或返回非 200**：检查是否遗漏了指引中的代理配置步骤。

## 0.3 注册账号 + 配置 SSH Key

**操作内容**：注册 GitHub 和 GitCode 账号，配置 SSH Key 实现免密 push / pull。

**操作步骤**：

1. 注册账号：

| 平台 | 地址 |
|------|------|
| GitHub | <https://github.com/> |
| GitCode | <https://gitcode.com/> |

2. 配置 git 全局身份与 SSH Key：

```bash
# 配置全局身份（邮箱必须与 GitHub 注册邮箱一致，否则 DCO 校验失败）
git config --global user.name  "Your Name"
git config --global user.email "your_name@example.com"

# 生成 SSH Key
ssh-keygen -t ed25519 -C "your_name@example.com"
cat ~/.ssh/id_ed25519.pub   # 复制输出内容
```

3. 将公钥分别上传到两个平台：
   - GitHub：<https://github.com/settings/keys> → New SSH key
   - GitCode：<https://gitcode.com/setting/key-ssh> → SSH 公钥

4. 验证连通性：

```bash
ssh -T git@github.com    # 期望: "Hi <your-name>! You've successfully authenticated..."
ssh -T git@gitcode.com
```

**易发问题**：

- **`ssh -T` 超时或被拒绝**：确认公钥已上传且选择了正确的 Key 类型（Authentication Key）；检查蓝区网络是否放行了 22 端口。
- **git config 邮箱填错**：邮箱不一致会导致后续 DCO 校验失败，修改方法：`git config --global user.email "正确邮箱"`。

---

## 0.4 签署 DCO

**操作内容**：了解并配置 DCO（Developer Certificate of Origin）签名，确保每次 commit 合规。

**操作步骤**：

每个 commit 需带 `Signed-off-by:` 行，使用 `-s` 参数自动追加：

```bash
git commit -s -m "feat: add fused softmax kernel"

# 如果漏了 -s，可以补签
git commit --amend -s --no-edit
```

> triton-ascend 当前要求 DCO 签名。其它项目（如 openEuler）可能要求 CLA，按 PR 里 bot 的提示链接签即可。

**易发问题**：

- **PR 的 DCO check 显示红灯**：通常是 commit 邮箱与 GitHub 注册邮箱不一致，或忘记加 `-s`。可以点击 "Details" 查看具体错误信息，按指引修改。

**操作内容**：确认 IDE 可用，完成插件配置，获得可立即开发的环境。

## 0.7 安装 pre-commit

**操作内容**：在项目中启用 pre-commit，把 CI 检查规则下沉到本地 commit 时刻，在 push 前发现格式和静态检查问题。

**操作步骤**：

1. 安装 pre-commit 并注册钩子（需先完成任务 Step 1 的仓库 clone，也可在 clone 后回来执行）：

```bash
pip install pre-commit
cd /workspace/triton-ascend
pre-commit install --install-hooks
pre-commit run --all-files                  # 首次运行，约 3–10 分钟
```

2. 参考蓝区 pre-commit portal 查询规则说明：🔗 <https://portal.libing.huawei.com/card/userFeedBack?hideSide=true&hideNotice=true&hideHelp=true&param=precommit>

**易发问题**：

- **`pre-commit run --all-files` 网络超时**：首次运行会下载 hook 工具，需要蓝区网络畅通。
- **hook 注册后 commit 报错**：先用 `pre-commit run` 查看具体哪条规则失败，再针对性修复。

## 🎯 任务：【全程通关】AI 写代码，提 PR 让名字出现在贡献者列表（约 120 分钟）

### 目标：用 AI agent + Skill 生成代码，完成本地验证，再走完 commit → push → PR → CI → Review 的完整链路。

### 我们要做什么

开源社区的贡献不只是写代码——你需要与社区沟通，把代码通过规范的流程送入主干。

今天要完成的是一次**真实贡献的完整旅程**：用 AI agent 加速代码生成，用 Skill 保证代码质量，最终提交一个被社区接受的 PR，让自己的名字出现在 triton-ascend 的贡献者列表中。

### 你将体验到的开发范式

| 环节 | AI 如何介入 |
|------|------------|
| 代码生成 | Agent 对话驱动，多轮交互生成实现骨架与单测 |
| 代码优化 | Skill 市场提供领域专属skills |
| PR 前自查 | 让 AI 扫一遍 diff，提前消解 Reviewer 会发现的问题 |

## 1.1 Fork 仓并建分支（约 10 分钟）

**操作内容**：Fork 上游仓，建立特性分支。

**操作步骤**：

1. 打开 [triton-ascend](https://github.com/triton-lang/triton-ascend)，点击右上角 **Fork**，fork 到自己账号下。
2. 进入自己的 fork 仓，点击左上角分支下拉框，输入新分支名，点击 **"Create branch: feat/xxx from main"**。
3. 将 fork 仓 clone 到本地并添加 upstream 远端：

```bash
git clone --recursive git@github.com:<your-name>/triton-ascend.git
cd triton-ascend
git checkout feat/xxx

git remote add upstream git@github.com:triton-lang/triton-ascend.git
git remote -v   # 确认同时有 origin（fork）和 upstream（官方）
```

**易发问题**：

- **🚫 永远不要在 fork 仓的 `main` 分支上直接改代码**——`main` 只用于与主干保持同步。

---

## 1.2 用 Agent 对话生成代码骨架（约 20 分钟）

**操作内容**：在 IDE 中打开 AI 助手（或直接使用对应Agent客户端），通过多轮对话生成算子实现骨架与单测模板。

**操作步骤**：

1. 在 IDE 中打开 AI 助手对话窗口。
2. 按以下三轮提示词流程与 Agent 对话：

```
# 第一轮：理解规格
"帮我分析XXX问题或XXX需求，提炼出需要实现的函数签名和输入输出约束"

# 第二轮：生成骨架
"基于以下规格，生成 Python 实现骨架和对应的 pytest 单测模板：
  - 函数名：xxx
  - 输入：...
  - 输出：..."

# 第三轮：细化
"上面的实现中，xxx 部分需要用 Triton kernel 写，帮我补全"
```

3. 核验 Agent 生成的产出文件是否完整（以新增算子为例）：

```
├── python/triton_ascend/ops/your_op.py   # 算子实现
├── test/ops/test_your_op.py              # 单元测试（必须）
└── docs/ops/your_op.md                   # 文档（必须）
```

**易发问题**：

- **Agent 输出内容与需求不符**：可能是上下文过长，点击"新建对话"重新开始，将 Issue 内容直接粘贴给 Agent。
- **Agent 生成的代码风格与项目不一致**：在提示词中加入"请参考项目中 `xxx.py` 的代码风格"，Agent 会读取现有代码作为参考。
- **AI 生成的代码不能直接打 `Signed-off-by`**——必须逐行阅读、理解、修改后，才算"你的贡献"，DCO 才合法。

## 1.4 本地构建 + 单测验证（约 10 分钟）

**操作内容**：在容器内完成增量编译，运行单元测试，确保代码在本地可以通过。

**操作步骤**：

示例：
```bash
# 增量编译
cmake --build build -j$(nproc)

# 运行当前算子的单测
pytest test/ops/test_your_op.py -v

# pre-commit 快速自检（仅检查改动文件）
pre-commit run
```

全绿 ✅ 后继续下一步；如有报错，先按提示修复。


## 1.5 pre-commit 全量自检（约 5 分钟）

**操作内容**：提 PR 前执行一次全量 pre-commit，确保所有文件均通过规则检查。

**操作步骤**：

```bash
pre-commit run --all-files
```

**易发问题**：

- **clang-tidy 误报，需要屏蔽某条规则**：按以下格式注释，机器人会自动 @ committer 审核：

```cpp
// NOLINT(rule-name) reason: 这里说明为什么必须屏蔽
```

- **pre-commit 全量耗时过长**：可以先 `pre-commit run --files <your-files>` 只检查改动文件，再 `--all-files` 做最终确认。

---

## 1.6 写规范 commit message 并提交（约 10 分钟）

**操作内容**：按 当前项目 Commits 规范写 commit message，完成本地提交。


**type 取值实例**：`feat | fix | docs | style | refactor | perf | test | chore | ci | build`

```bash
git add python/triton_ascend/ops/your_op.py \
        test/ops/test_your_op.py \
        docs/ops/your_op.md
git commit -s    # -s 自动追加 Signed-off-by，编辑器中填写完整 message
```

**易发问题**：

- **commit message 被 pre-commit 拦截**：通常是标题超过 50 字符、缺少 type 前缀或缺少 `Signed-off-by`，按提示修改后重新 commit。
- **反例**（不要这么写）：`update`、`fix bug`、`按 review 意见修改`——这些没有传递有效信息。

---

## 1.7 Push 并提交 Pull Request（约 10 分钟）

**操作内容**：将本地分支推送到 fork 仓，在 GitHub 上创建 PR 并填写规范描述。

**操作步骤**：

```bash
git push origin feat/xxx
```

1. 打开 fork 仓页面，点击黄色提示条中的 **"Compare & pull request"**。
2. 确认目标分支是 `triton-lang/triton-ascend:main`。
3. 按社区pr模板填写 PR 描述。

**易发问题**：

- **PR 改动量 > 1000 行**：强烈建议拆分为多个 PR，每个 PR 只做一件事，便于 Review 和回退。

---

## 1.8 关注 CI 流水线（约 15 分钟）

**操作内容**：监控 PR 触发的 CI 流水线，处理红灯。

**操作步骤**：

PR 提交后会自动触发流水线。（部分项目需要committer帮助触发CI）

红灯处理流程：

```bash
# 1. 点击 PR 页面查看具体失败日志
# 2. 本地复现 + 修复后
git add <fixed-files>
git commit -s -m "fix: xxx"
git push --force-with-lease origin feat/xxx   # --force-with-lease 比 --force 安全
```

**易发问题**：

- **`--force-with-lease` 被拒绝**：说明远端分支被他人修改过，先 `git fetch origin` 再决定如何处理。

---

## 1.9 响应 Review，等待合入（约 15 分钟）

**操作内容**：处理 AI 辅助检视和人工 Review 意见，直到代码合入主干。

**操作步骤**：

评审通常包含两层：
1. **AI 辅助检视**：优先处理 high-severity 项；低误报可在评论中说明原因。
2. **人工 Reviewer**：通常需要 2 个 Reviewer + 1 个 Committer 批准。

| 场景 | 推荐做法 |
|------|----------|
| 同意意见 | 修改后回复 "Done in commit `<sha>`"，并 @ 评审人 |
| 有不同意见 | 给出技术理由 + 数据 + 引用文档；不要情绪化 |
| 评论已解决 | **让 Reviewer 自己 resolve**，不要自行 resolve |
| PR 与主干冲突 | `git fetch upstream && git rebase upstream/main`，解决后 force-push |

通过所有检查后，Committer 会执行合入。合入后：

- ✅ 关联 Issue 自动关闭
- ✅ 你的头像出现在 contributors 列表 🎉

**易发问题**：

- **Reviewer 长时间不响应**：在 PR 中礼貌 `@xxx PTAL`（Please Take A Look），或在社区周会上提一下。
- **PR 与主干冲突**：执行 `git rebase upstream/main` 解决冲突，建议不用 `git merge`（会产生多余的 merge commit）。
